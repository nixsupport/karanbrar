About

A glimpse at my professional journey so far...

  • 2005-2009

    Initial Struggle and Learning

    Started my career in 2005 as a Linux and Network Administrator. Although I had certifications like RHCE and CCNA, working in real world was totally a different experience. I started off with LAMP based environments where I had to troubleshoot configurations of Apache and MySQL in addition to Linux. At times I also had to dig in to PHP code to troubleshoot issues. That helped to build an overall understanding about web apps running on Linux environments. I struggled a lot but I'm happy with the reward of that struggle I did. That helped to build a strong foundation for my future.

  • 2009

    First Face Off with AWS Cloud

    I got a chance to work on AWS in 2009 and the first project I built was a postfix cluster running on bunch of EC2 instances. Believe me it was so much hard to manage EC2 instances running with only ephemeral storage. However, it was a unique experience. Instead of setting up new physical boxes, I was able to spin a VM in a few minutes. That was fascinating.

  • 2011

    Got a Big Break

    I joined Emerson Electric Co. in 2011 where I spent 7.5 years of my career. I learned a lot, not just new technologies but also other aspects of life. A big shout out to my manager and mentor Mike Mantle for his continuous support. I got a chance to work on Middlware projects where I setup clusters of Oracle WebLogic Server supporting SOA, OSB and ADF projects. I also setup IBM MQ servers and build CI/CD pipelines to automate the deployment of these projects.

  • 2011-2017

    Evolved as a Cloud Architect

    I worked with many clients specifically on AWS. Every project was a new challenge with some custom requirements. That helped to sharpen my skills and start architecting solutions effectively. Being an Architect, you have to visualize different components working together and then should also know how to stitch those components. A unique requirement everytime helped me to understand the AWS cloud eco system in depth.

  • 2018

    Moved to Canada

    It was my personal milestone achieved in 2018 when I moved to Toronto. It's been great to see people of different religions and beliefs in such a diverse environment. I got some exciting opportunities here which gave me a boost to excel in my career.

Kubernetes Dashboard SSO

This solution was implemented to access Kubernetes Dashboard via OpenID provider in Amazon EKS. There is no out of the box solution exist. It was achieved using openresty proxy deployed in EKS.

  • Client: HOOPP
  • Date: September 2019
  • Service: DevOps

EMR and Step Functions

This solution was designed for ETL workflow. AWS StepFunction service was used to orchestrate the workflow. It gets triggered by a time based CloudWatch event rule. Once trigger, initial scripts are executed on EC2. On successful execution, a parallel job started within StepFunction. As a result of which EMR cluster is launched and then Step are executed. Appropriate status checks are implemented with "Wait" tasks in Step Functions.

  • Client: HOOPP
  • Date: April 2019
  • Service: BigData

Docker-In-Docker

This solution consolidates the build agents required for Azure DevOps Build and Release pipelines. Agents run inside EKS with docker installed in order to communicate with ECR. Build Agents running in EKS have docker container running as a side-car in same pod so that instead of mounting EKS host volume for docker client, it is actually referred to side container over TCP.

  • Client: HOOPP
  • Date: February 2019
  • Service: DevOps

Detect & Remove CloudFormation Drift

This solution provisions IAM roles and policies using CloudFormation templates. In addition to that, if anyone manually edits the role, the drift will be detected and changes will be removed automatically.

  • Client: HOOPP
  • Date: January 2019
  • Service: Detect & Remove CloudFormation Drift

Disaster Recovery

This solution was implemented to automated the DR process. Daily EBS snapshots being taken everyday using AWS Backup service. All EBS volumes are tagged with mount point and instance id. In the event of failure, a new instance is launched, all the snapshots for original instance are pulled from Backup service, volumes are created and attached with the new instance based on the mount point tags.

  • Client: HOOPP
  • Date: December 2018
  • Service: Cloud Architect

Infrastructure Pipeline

This implementation allows deployment of CloudFormation templates using CodePipeline. Engineers check-in the templates in CodeCommit which triggers the Pipeline. In Build stage, template syntax is validated using cfn-lint followed by security validation using cfn-nag. It reports if permissions are too permissive and fails the build in that case. Next stage is called Approval where anyone from designed set of individuals reviews and approves the change. In Deploy stage, templates are pushed to S3 first and then deployed using CloudFormation as Nested Stacks.

  • Client: iAdvisory
  • Date: November 2018
  • Service: DevOps
